Architecture and Performance Report

Ray In-Memory Backend for DataDesigner

This report summarizes the implementation, architectural decisions, benchmark results, and remaining optimization work for adding an optional Ray-backed execution path that accepts Ray-resident inputs and returns Ray-resident outputs without changing default local behavior.

worktree: /Users/eric/src/DataDesigner-ray-backend branch: feat/ray-backend date: 2026-04-23 Ray: 2.55.1 Python: 3.13

Executive Summary

Implemented

A new optional RayBackend execution path plugs into DataDesigner through a small backend protocol. Users can pass a Ray Dataset or Ray ObjectRef sequence as input_dataset and receive either a Ray Dataset or Arrow ObjectRefs back.

Default Behavior

The local path remains the default. Ray is imported lazily and is only installed through data-designer[ray]. Without a backend, create() keeps the existing local artifact-backed behavior.

Performance Takeaway

Ray is slower for tiny cheap jobs because setup overhead dominates. For API-bound LLM jobs, Ray is already competitive locally and should improve with larger block sizes, real clusters, and worker/client reuse.

Important fix from benchmarking: running local seeded jobs before Ray jobs initially exposed a serialization failure from cached DuckDB connections inside seed readers. The Ray backend now clones seed readers into clean unattached worker copies before sending them to Ray workers.

Layering and Modularization

The implementation preserves the repository's structural layering: interface depends on engine, engine depends on config, and config remains independent of both. Ray is an optional integration owned by the interface package, not a dependency of config or engine.

Execution Architecture

The local path still compiles config, builds artifacts, profiles the output, and returns a local DatasetCreationResults. The Ray path converts the input data plane into Ray Data blocks, executes each block with worker-local engine resources, and keeps the result in Ray.

Ray Worker Boundary

The most important architectural boundary is the worker payload. Anything sent through Ray must be serializable and should be data-like. The backend now copies only the necessary interface state and removes attachment state from seed readers before serialization.

Serialization rule: worker options must contain plain config/state, not open handles. The benchmark found DuckDB connections retained inside seed readers. The fix clones readers and clears source, secret_resolver, and _duckdb_conn before Ray sees them.

Public API Shape

The intended user path is simple: existing local users do nothing. Ray users install the extra, initialize or connect to Ray, and pass the backend plus an in-memory Ray input handle. 

import ray
from data_designer.interface.data_designer import DataDesigner
from data_designer.integrations.ray import RayBackend

ray.init(address="auto")  # or an existing local/session cluster

designer = DataDesigner(
    artifact_path="artifacts",
    model_providers=[...],
    backend=RayBackend(batch_size=1024),
)

results = designer.create(config_builder, input_dataset=ray_dataset)
ray_dataset_out = results.load_dataset()

Supported input forms are Ray Dataset, Arrow ObjectRefs, and pandas DataFrame ObjectRefs. Supported output forms are Ray Dataset and Arrow ObjectRefs.

What We Learned

Ray helps when work is large enough

For cheap sampler/expression jobs, Ray overhead is too high. For LLM jobs, model latency dominates enough that Ray is close even on a local instance. On a real cluster and with larger record counts, the Ray path should become more attractive.

State boundaries matter

DataDesigner has stateful readers and resource providers optimized for a local process. Ray forces us to separate configuration from execution state. The seed-reader clone fix is the first concrete example of the right architectural direction.

Returning refs works

The Ray backend can return Arrow ObjectRefs, so the output can remain in Ray object store instead of materializing on the driver. This is essential for large-data jobs.

Order semantics differ

Ray Data does not guarantee the same output ordering as the local path under all execution plans. If stable order is required, the backend should inject and preserve an internal row id.

Current Limitations

  1. Per-block setup cost: each Ray block currently creates temp artifact storage, resource providers, registries, and a DatasetBuilder.
  2. Per-block model health checks: workers may repeat checks that should be a driver preflight or cached per worker.
  3. No actor reuse: map_batches uses a function. A callable class or actor-style execution could reuse model clients and resource setup across blocks.
  4. Local profiler gap: Ray results are Ray-resident and load_analysis() currently returns None. Profiling needs a distributed/Ray-native implementation.
  5. Metrics aggregation gap: worker model-usage stats are not yet aggregated into the Ray result object.
  6. Ordering: output row order can differ from input order unless explicitly preserved.
  7. From-scratch partitioning: ray.data.range(num_records) needs explicit block planning for high throughput.
  8. Processor side effects: processors that write side artifacts were built for local storage and need clearer Ray semantics.
  9. Secrets and trust boundary: provider config and secret resolvers are serialized to workers. That is normal for trusted Ray clusters, but should be documented.

Optimization Roadmap

Phase 1: Correctness Hardening

  • Add stable row-id preservation for Ray inputs and from-scratch generation.
  • Reject or document processors with unsupported distributed side effects.
  • Preflight model health checks on the driver and skip repeated worker checks.
  • Add tests for ordering, ObjectRef input formats, and same-process local-then-Ray execution.

Phase 2: Worker Reuse

  • Use Ray Data callable classes for map_batches so setup runs once per worker.
  • Reuse model client pools and resource providers per worker.
  • Replace temp artifact storage with an in-memory preview artifact abstraction where possible.
  • Broadcast immutable compiled configs instead of deep-copying builders repeatedly.

Phase 3: Cluster Throughput

  • Expose override_num_blocks / block planning controls.
  • Coordinate provider throttling globally across Ray workers.
  • Support actor placement, CPU/GPU/resource labels, and autoscaling-friendly defaults.
  • Add Ray-native usage metrics and trace collection.

Recommended Next Implementation Steps

  1. Promote the benchmark into the repo. Move the temporary benchmark into scripts/benchmarks/, parameterize provider/model/record counts, and emit JSON plus HTML.
  2. Make block planning explicit. Add num_blocks or target_block_size options to RayBackend and use Ray APIs to avoid accidental tiny blocks.
  3. Switch to worker setup reuse. Implement a Ray Data callable class that constructs model clients and DataDesigner resources once per worker process.
  4. Add a distributed metrics channel. Return or collect usage stats, task traces, and error summaries from workers into RayDatasetCreationResults.
  5. Define order semantics. Either document unordered output or preserve a hidden row id and optionally sort on output.

Bottom Line

The architecture is now in the right shape: Ray is an optional interface-level backend, not a cross-cutting dependency, and the engine remains reusable through an explicit async execution selector. The current Ray path proves the in-memory contract end to end, but it is still a first functional backend, not an optimized distributed runtime. The highest-leverage next step is worker reuse, followed by explicit partition planning and distributed metrics aggregation.
